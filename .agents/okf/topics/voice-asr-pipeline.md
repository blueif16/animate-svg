# Why / how it works (the lifecycle, end to end)
`w3a-voice-asr` generates the per-cue audio truth: `@studio/narration-kit`'s `generate-voice.mjs --align`
(external, vendored — same sibling-package boundary as [[timeline-reconcile]]) synthesizes+trims one WAV clip
per cue from w2b's `script-cues.json`, emitting the frozen `src/lessons/generated/<CamelId>Clips.ts` array
(`ClipCue[]`, each entry's `narrationFrames` exact-measured at 30fps — "DO NOT EDIT BY HAND"). Locally-owned,
in-repo code sits AFTER that generation step as a deterministic quality gate: `lesson-audio-gate.mjs` runs three
HARD checks (held-vowel drone / untrimmed dead-air / empty-short clip) that block on failure, plus one ADVISORY
truncation/coverage check that shells out to `asr-clip-coverage.py` (a per-clip ASR sidecar, generation-
independent of which TTS produced the audio) and never blocks. This is the only node allowed to write voice
audio + the generated Clips/Timing files.
