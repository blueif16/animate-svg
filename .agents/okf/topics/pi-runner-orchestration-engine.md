# Why / how it works (the lifecycle, end to end)
`.claude/workflows/lesson-build.js` is the SOURCE OF TRUTH: a Claude Code Workflow file (`export const meta` +
phase prose, wave order, `args.lessonId`/`startAt`/`style` params) authored and PROVEN the normal way — by
spawning the real Claude Code Workflow tool. `pi-runner` does not re-define the graph; it EXECUTES that same
file under recording stubs. `extract.mjs`'s `extractMeta()` parses the file's `meta` export (title/phases), and
`extractWorkflow()` runs the file's body with the Workflow runtime hooks (`agent`/`parallel`/`phase`/`log`)
stubbed to RECORD rather than execute — capturing the exact realized prompts + stage DAG for the given
`args`, so editing the wave prose in `lesson-build.js` propagates with no hand-sync and no codegen. `run.mjs`'s
top-level driver loop calls `extractWorkflow()` once, then walks `stages` (parallel lanes grouped by phase),
calling `runNodeWithEscalation()` (line 1226 — wraps `runNode()` with a consult-on-failure escalation path) for
every node in a stage via `Promise.all`, halting the whole run if any node returns `error`/`blocked`.
`runNode()` (the ~450-line core) builds the per-node `pi` CLI invocation (`piArgs()`), the sandbox profile
(`buildSandboxProfile()` — scopes filesystem access per node), schema-validates the node's returned JSON
(`schemaCheck()`), and enforces the node only wrote within its declared owned globs (`withinOwned()`). The
driver owns determinism (stage order, parallel lanes, status, watchdog, worktree isolation via
`setupWorktree()`/`finishWorktree()` when `--worktree` is passed); `pi` (a non-Claude coding-plan model) owns
each node's actual read/bash/edit/write work. `watch.mjs`/`status.mjs` are the polling monitors Claude Code (the
console) uses to follow a run without touching it.
