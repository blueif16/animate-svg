# Why / how it works (the lifecycle, end to end)
Two independent preview surfaces both derive from [[primitive-registry]]'s `primitive-registry.json` ‚Äî neither
invents its own data model, both are read-only projections a human uses to visually QA a primitive. The STATIC
gallery: `build-gallery-data.mjs` flattens the registry's sections into one card list (tagging each by its
existing kind/family/type ‚Äî `useWhen`/`intent`/`avoidWhen`/`variants`/`component`/`source`/`status`/
`supersededBy`, reading no new fields) plus the generated `IconAsset` SVGs under `public/icons/`, writing
`gallery/gallery-data.json`; `render-previews.mjs`'s `renderOne()` then bundles+renders a short preview clip per
card (used to catch a GIF/preview that never actually moves ‚Äî `gifHasMotion()`); `serve.mjs` serves the static
`gallery/index.html` + data locally (`npm run gallery`). The LIVE-Studio surface is separate:
`ComponentGallery.tsx`'s `ComponentGallery` component lays out every registered component in one scrollable
Remotion composition (own still-frame/duration constants) for browsing inside `remotion studio`;
`PreviewComposition.tsx`'s `PreviewComposition` renders ONE component in isolation at a fixed
760x440/90-frame/30fps preview size (`fit` vs `size` mode) with a poster frame, and `PreviewRoot.tsx` is the
Studio registration entry point wiring both into Remotion's composition list. `demoProps.tsx` supplies the
placeholder prop sets each gallery card renders with.
