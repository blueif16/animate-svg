# capability-registry — map

_The drift-gated single source of truth for "what reusable capabilities exist." Free-form, no scores. Refresh when the gate set or the prose homes change. Registry row: `.agents/tracked-systems.md#capability-registry`._

**Scope reminder:** we optimize the **prose** (item `intent`/`avoidWhen`/status, `CAPABILITIES.md` discipline, the harness skill rules). The gate **scripts** are wiring — out of Hermes edit-scope.

## What it is / what orchestrates it
- **Not a wave workflow — a GATE + a corpus.** Driven by `npm run registry:check` (6 gates) wired into pre-commit. The corpus is the truth; the gate keeps the JSON `== ` the code.
- **Source of truth:** `remotion-svg-primitives/src/capabilities/primitive-registry.json` (code-as-truth, generated by `build-registry.mjs` from the actual exports) + the hand-authored prose `.agents/CAPABILITIES.md` + the generated agent digest `catalog-digest.md`.

## The gate scripts (wiring — read to diagnose, don't edit here)
`scripts/registry/`: `build-registry.mjs` (sweep code → JSON), `catalog-digest.mjs` (JSON → agent menu), `drift-report.mjs`, `check-deprecated-refs.mjs` (live→deprecated reference fails), `check-lesson-primitives.mjs` (a lesson citing a deprecated/absent id fails — liveness, not just membership), `code-unions.mjs`, `families.mjs` (one-module-per-shared-fact), `export-schema.mjs`, `check-gallery.mjs` (every component needs `demoProps`).

## What we optimize (the markdown surfaces — the owners routing points to)
| Surface | Owns | Optimize when |
|---|---|---|
| each LIVE item's `intent` sentence + `avoidWhen` | how an agent SELECTS the right component (the digest's disambiguators) | a sibling wrong-pick or a duplicate-build traces to thin/ambiguous prose |
| item `status` + `supersededBy` | deprecation hygiene (a deprecated id must be invisible + uncitable) | a stale/deprecated id leaks to an agent |
| `CAPABILITIES.md` §A (creation rules, sizing clause) | the bar a capability must meet AT creation (lesson-agnostic, prop-driven, default-size = good-size) | a new capability ships flat/ugly/topic-baked |
| `capability-registry-harness` SKILL | the gate intent + the add-protocol | the gate misses a drift class or over-fires |

## Wiring — who relies on this
- **lesson-build** W2a/W3b/W4a read `catalog-digest.md` to pick components; W3b writes new registry entries.
- **capability-gap-filler** Build node writes entries; Verify runs `registry:check`.
- **capability-gallery** is a thin view over `primitive-registry.json` (no parallel data model).

## Governing docs
`CLAUDE.md` "## Capabilities" + "Generated decorative SVG assets" (visual-source order: reuse → parametric primitive → generate). `CAPABILITIES.md` is the hand-authored prose half; the JSON is the generated half.

## Observability (where a flaw shows)
`npm run registry:check` output (the 6 gates + count documented), `drift-report.mjs`, the generated `catalog-digest.md` (is it a usable menu?), the gallery (does the item look right?).
