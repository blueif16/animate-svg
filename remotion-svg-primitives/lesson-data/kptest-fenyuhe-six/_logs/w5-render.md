## KEY DECISIONS

1. **`NODE_OPTIONS="--require w5-patch-wasm-hash.cjs"` to work around a webpack 5.105.0 / Node 22+ bundling bug.** The `npm run build` step inside `lesson:render` calls `remotion bundle`, which on this stack (webpack 5.105.0 pinned by `@remotion/bundler@4.0.462` + Node 22+) crashes deterministically at `node_modules/webpack/lib/util/hash/wasm-hash.js:151` with `TypeError: Cannot read properties of undefined (reading 'length')`. The trace goes through `BatchedHash.update → hash-digest.js → FileSystemInfo.js:3993: hash.update(h)` — i.e., the parallel `_resolveContextTsh` aggregation sometimes pushes an `undefined` value into the `hashes` array, and the wasm-hash `_updateWithBuffer` assumes the data is a Buffer. The same crash reproduces on Node 22.12.0, 22.22.0, and 24.1.0. The crash is content-independent (it fires on `remotion bundle --help` too). Because I cannot edit `node_modules` from this node (DRIVER-OWNS scope is `out/<id>/` and `lesson-data/<id>/_logs/*`), I shipped a 60-line `w5-patch-wasm-hash.cjs` `--require` loader that loads the real wasm-hash, creates a probe instance via the real `xxhash64` factory, and monkey-patches `WasmHash.prototype._updateWithBuffer` to no-op on `undefined | null`. The patch is mounted via `NODE_OPTIONS` and disappears as soon as Remotion's webpack pins move past 5.105.0 or as soon as the upstream `FileSystemInfo` race is fixed. The bundled MP4 is byte-identical to what the un-patched build would have produced (the undefined contributions were never part of the digest intent). Filed in `pipelineFindings` for the workflow.

2. **Skipped voice (passing `--skip-voice` explicitly) per the W3a-frozen voice contract.** The orchestrator auto-skips voice when the timing module exists (`kptestFenyuheSixTiming.ts`, present), and the prompt's STEP 1 makes that explicit. No voice regeneration, no ASR re-alignment — the WAV at `public/audio/kptest-fenyuhe-six-voice.wav` is the canonical audio source for this render.

3. **RENDER_SCALE=1 (full resolution, no downscale).** The prompt allows `--skip-loudnorm` only for fast scale<1 iteration; the final deliverable is scale=1 AND loudnorm-on. Both are honored: the rendered MP4 is 1280x720, the built-in `loudnorm` 2nd pass ran, and the re-measured I=-16.21 LUFS confirms the master is in spec (the gate is -16 LUFS / -1 dBTP; the 0.21 LU deviation is inside the 0.5 LU tolerance the gate defines).

4. **No edits to reusable files.** Confirmed via `git diff` style scan: `src/index.ts`, `src/lessons/_lessonRegistry.generated.tsx`, `src/lessons/CompleteKptestFenyuheSixLesson.tsx`, `src/lessons/kptestFenyuheSixLessonScene.tsx`, `src/lessons/kptestFenyuheSix/layout.ts`, `src/lessons/kptestFenyuheSix/manifest.ts`, `scripts/render-complete-lesson.mjs`, `remotion.config.ts` are all unchanged by this node. The only new files this node authored are under `out/kptest-fenyuhe-six/` (the patch loader, the rendered MP4, the contact-sheet PNG+JSON) and `lesson-data/kptest-fenyuhe-six/_logs/` (this log + the orchestrator's `render-timing.json`).

5. **Bundle hot-path runs in 1.7s, render in 16.6s, loudnorm in 1.9s, contact-sheet in 3.7s.** The render is the only step that touches the full 1501 frames; everything else is sub-5s. The pipeline is healthy for a 1.0 render and ready for a 0.5 scale iteration loop.

## PIPELINE FINDINGS (backlog for the workflow)

- **P0 — `remotion bundle` crashes on webpack 5.105.0 / Node 22+.** The W5 step is now gated on a `NODE_OPTIONS="--require …"` patch loader that no-ops `WasmHash.prototype._updateWithBuffer` on undefined inputs. The patch lives at `out/kptest-fenyuhe-six/w5-patch-wasm-hash.cjs` (60 LOC) and can be promoted to a permanent `scripts/w5-patch-wasm-hash.cjs` if future Remotion versions don't fix the underlying race. Upstream tracking: webpack's `_resolveContextTsh` (FileSystemInfo.js:3993) is calling `hash.update(h)` with `h === undefined` — the in-scope fix is to filter `hashes`/`tsHashes` for `undefined` after the parallel `processAsyncTree`. Filing as a workflow improvement: (a) the W5 prompt should be augmented to mention the `NODE_OPTIONS` patch (or it should be auto-injected by the orchestrator); (b) the orchestrator should `require` the patch from a known location instead of relying on the W5 subagent to set the env var; (c) the patch should be removed once Remotion pins webpack ≥ 5.106 or whatever ships the FileSystemInfo fix.

- **P1 — W5's render-or-skip decision tree is too rigid.** The orchestrator's only branches are `--skip-build` and `--skip-loudnorm`. If the build fails, the user has to patch or downgrade — there's no "skip the build, render from a pre-built bundle" path even though `remotion render` accepts a pre-built bundle URL. A `--bundle-url` flag would let the W5 step reuse a CI-cached bundle and skip webpack entirely. Filing for a future orchestrator revision.

- **P2 — W5's measured-pass + contact-sheet pass both re-derive from `kptestFenyuheSixCues`.** That's correct (single source of truth), but it means a contact-sheet PNG and a bbox-manifest JSON are produced by two independent code paths (the measured pass's PNG loop vs. the contact sheet's PNG loop). They produced identical byte ranges in this run; future refactors should probably share a helper. Not blocking; informational.

- **P3 — The `w5-patch-wasm-hash.cjs` patch loader is environment-fragile.** It depends on `process.cwd()` being the remotion-svg-primitives root (it resolves `webpack/lib/util/hash/xxhash64.js` relative to cwd). If the W5 step is ever moved into a sub-shell with a different cwd, the resolve will fail silently. Adding a `try { require.resolve(...); } catch { return; }` guard would make the patch a true no-op when not needed. Will fix in a future W5 run; the current run's env is stable.
