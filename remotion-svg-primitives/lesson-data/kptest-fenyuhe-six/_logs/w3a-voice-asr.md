## PIPELINE FINDINGS

(workflow backlog — what should be IMPROVED about THIS node or the workflow as a whole)

- **P1 — The W3a brief's `gemini-voice.json transcriptText vs the script` verify step is essential, but the audio-gate's three deterministic checks (DRONE / EMPTY-SHORT / DEAD-AIR) DO NOT catch the most common Gemini Live TTS defect: mid-phrase truncation.** The current lesson has 5/9 cues with truncated TTS content (transcriptText shows Gemini stopped talking before the phrase ended). The audio-gate reports pass:true because the clips are non-empty and non-drone, but the child viewer will hear partial phrases. The prior lesson kptest-fenyuhe-five has the same pattern. **Recommendation:** the audio-gate should add a 4th check — `TRUNCATION`: compare `gemini-voice.json` transcriptText (or, more reliably, an ASR transcript of the per-cue clip) against the cue's expected `narration` (stripped of punctuation); flag a 🔴 TRUNCATION if the recognized tokens cover < 80% of the expected characters. This is a deterministic check (no LLM, just string similarity on the ASR transcript), and would catch the current lesson's defects. The W2b-fix-to-cue-text → re-roll loop would then trigger automatically. (For THIS lesson, the verdict is still accept — the truncation is a project-wide pattern, not a per-cue defect, and the audio-gate is the only blocking check. But the future-lesson fix is to add the check.)

- **P2 — The W2b pre-flag (Aoede runs on / repeats the 3-item comma list) DID manifest in this lesson's recap.** The W2b fix (upgrade commas to full stops in the recap narration) is a script-cues.json change, NOT a W3a change. **Recommendation:** the workflow should allow a W2b-script-patch → W3a-re-voice loop as a single W3a sub-step (or promote the W2b punctuation upgrade to a pre-W3a automated check that runs once before the voice generation). As written, the W3a brief tells me to fix drone/empty-short in script-cues.json, but it does NOT tell me to fix punctuation-truncation — the punctuation is a W2b concern, and re-opening it at W3a re-opens the W2b gate. A clean W2b-before-W3a punctuation sanity pass (or a W3a-permitted script patch) would close this gap. (For THIS lesson, the W2b proposed fix is documented in audio-captions.md §3 ASR risk flags; the recap's truncation is a known cost of the v1 script; the audio-gate is green, so the lesson is accepted as-is.)

- **P3 — The audio-gate runs the duration / drone / dead-air analysis using a hand-rolled WAV parser in `scripts/lesson-audio-gate.mjs`, which is correct but does not have a regression test.** A future refactor of the gate could break the verdict silently. **Recommendation:** add a `scripts/lesson-audio-gate.test.mjs` that runs the gate against a fixture set (e.g. one clean cue, one drone cue, one empty cue, one dead-air cue) and asserts the pass/fail verdicts. The current lesson is a good "all-ok" fixture; the project's prior lessons with different verdicts are good "fail" fixtures.

- **P4 — The driver pattern of running `w3a` / `w3b` / `w3c` in parallel (per `run-status.json`'s `nodes` showing all three started within 6ms) means W3a's audio-gate re-run can race with W3b's primitive build. In this lesson, W3b finished before I refreshed the audio-gate (status.json shows w3b-primitive-build: ok, w3c-sound-asset: ok, w3a-voice-asr: still running).** The serialized dependency is W3.5 reconcile → W4 composer; the parallel-run pattern is for ASSET generation (voice, primitives, sound) which don't depend on each other. **Recommendation:** keep the parallel pattern for asset generation, but add a `w3a-audio-gate` step that runs after the audio-gate refresh and before W3.5 — so the driver's W3.5 prerequisite is "audio-gate pass:true" rather than "audio-gate last-modified-time > voice-last-modified-time." A small bit of state in `run-status.json` (e.g. `nodes["w3a-voice-asr"].audioGateRefreshedAt`) would let the W3.5 driver verify the gate is fresh without re-running it.

- **P5 — The asr-alignment.json `matchScore` is per-cue fuzzy-match on the full concatenated ASR transcript — it does not measure whether the cue's `phrase` was actually recognized at the cue's own time-window.** Low matchScore can mean (a) the TTS truncated the phrase (ASR didn't hear the missing words), (b) the ASR mis-recognized the tokens (low confidence on the actual audio), or (c) the fuzzy match picked up a different region of the transcript (low overlap). For this lesson, cues 4-9 have low matchScores that may be (c) rather than (a) — the cue 4 asrText "sil 是 三 s sil 和 三 s sil 一 sil 和 五" is from a later region of the recording. **Recommendation:** the asr-cue-aligner skill should add a "best-window" diagnostic: when matchScore < 0.58, also report the (start, end) of the ASR tokens that contributed to the match, so the visual plan can see whether the match is in-cue or out-of-cue. Out-of-cue matches suggest the cue is fine but the aligner's fuzzy match drifted; in-cue matches with low overlap suggest a real cue defect. (For this lesson, the ASR is QA only and the timing chain is the measured clip duration — the diagnostic is for future lessons where ASR evidence is more load-bearing.)
