# Project Rules

## Entry Point

To make a lesson video, write **one file**: `lesson-data/<lesson-id>/brief.md`. Then run the pipeline. `brief.md` is the only lesson-specific input ŌĆö everything else (wave order, kids-eye blocks, frame-literal rules, voice audit, reporting) lives in CLAUDE.md and the skills.

**Brief schema.** Seven sections, all unique info; trim or extend as needed but keep it minimal. **Note:** the `Length` field is a HINT for scope-setting, not a contract. The actual lesson length emerges from Wave 3.5's reconcile (`sum(max(narration, motion) + tail)`). Brief targets that don't survive contact with real narration are accepted as drift ŌĆö see `docs/pipeline-architecture.md`.

```markdown
# <lesson-id>

**Audience.** <age, language>
**Length.** <seconds band ŌĆö HINT for scope, not a target the composer pads to>
**Builds on.** <prior lesson ids | none>
**Style.** <style-id | default>

## Knowledge point
<one paragraph: what the child leaves knowing>

## The one beat
<the central demonstration this video exists for ŌĆö concrete, visual>

## Out of scope
<what NOT to teach; boundary with adjacent KPs>

## Continuity
<reuse what from prior lessons; identity-invariant promises>

## Narration notes
<known TTS risks, precedent fixes ŌĆö optional>
```

**Naming rule.** Lesson IDs are kebab-case with a curriculum-position prefix so they never collide: `kp<n>-<semantic>` (e.g. `kp2-counting-by-tens`, `kp3-tens-and-ones-place`). The prefix flows everywhere ŌĆö folder, MP4, voice `.wav`, generated timing module, and React composition name (`CompleteKp2CountingByTensLesson`).

**Folder rule.** Inputs and generated outputs are split into two top-level locations, keyed by lesson id.

- **Inputs ŌĆö `lesson-data/<lesson-id>/`** (authored, hand-edited, version-controlled): `brief.md`, `pedagogy.md`, `storyboard.md`, `visual-design.md`, `audio-captions.md`, `sketch-overlay.md`, `primitive-gap-scan.md`, `verification.md`, `script-cues.json`, and the scaffolded `pipeline.json`. Nothing the pipeline writes lands here.
- **Outputs ŌĆö `out/<lesson-id>/`** (generated, derivable, can be `.gitignore`d): `<lesson-id>.mp4`, `<lesson-id>-contact.png` + `<lesson-id>-contact.json`, `gemini-voice.json`, `asr-alignment.json`, `bbox-manifest.json`, `primitive-checks/`, `verification-frames/`. `outDir` is derived mechanically from `lessonId` ŌĆö pipeline.json does not configure it per lesson.
- **Two technical exceptions**, dictated by Remotion/TS imports:
  - Voice `.wav` stays at `public/audio/<lesson-id>-voice.wav` ŌĆö Remotion serves static media from `public/`.
  - Generated timing modules stay at `src/lessons/generated/<camelLessonId>Timing.ts` and `<camelLessonId>Silences.ts` ŌĆö they are TypeScript modules imported by scene code; the Remotion bundler resolves them via TS paths and moving them under `out/` breaks the build.

**Commands.**

```bash
cd remotion-svg-primitives && npm run lesson:scaffold -- --id <lesson-id>   # creates pipeline.json
cd remotion-svg-primitives && npm run lesson:render   -- --config lesson-data/<lesson-id>/pipeline.json
```

`pipeline.json` is mechanical (paths + voice config) and is generated by scaffold ŌĆö do not edit by hand. Reusable scripts, primitives, and components stay lesson-agnostic ŌĆö never hardcode lesson topics, copy, timings, or paths into them.

Use `-- --skip-voice` only when intentionally reusing an existing aligned voice file. (The render script auto-skips voice if the generated timing module already exists for the lesson.)

ASR alignment is an owned production artifact, not a pass/fail gate. Inspect the detected transcript, token events, timestamps, and tokenizer settings, then correct cue timing from evidence.

## Subagent Workflow (wave order)

The entire loop ŌĆö every wave below, **including pedagogy (Wave 0) and reconcile (Wave 3.5)** ŌĆö is owned by **one self-contained workflow** (the `Workflow` tool), parameterized by the lesson knowledge and runnable in parallel across many lessons. The orchestrator's only job is to spawn that workflow with the knowledge and observe; it implements nothing and owns no wave directly. Each wave is a clean-room subagent given concrete inputs. Invoking a single wave's subagent by hand is a **debug-only exception** ŌĆö taken at the *start* of a run while improving the system, never the default. See `docs/pipeline-architecture.md` for the timeline-coordination architecture (cue-as-unit, frozen audio, Wave 3.5 reconcile).

0. **pedagogy** (first workflow node) ŌĆö write `lesson-data/<id>/pedagogy.md` answering for each cue: what does the child discover here that they didn't know walking in? Follow `lesson-pedagogy` skill. No downstream wave advances without this file.
1. **storyboard** alone ŌĆö reads `pedagogy.md`; defines cue IDs, narration beats, required visuals. NO duration estimates.
2a. **visual-design** alone (SERIAL ŌĆö must finish before 2b) ŌĆö Visual Contract with zones, identity-invariant, **per-cue `visualMotionSeconds`** (the motion budget). Intent only, no absolute frames. Every row references a pedagogy discovery.
2b. **audio/captions** (depends on 2a) ŌĆö writes narration TO FIT each cue's visualMotionSeconds at the calibrated voice rate (~0.30s/Chinese-char for Aoede). NO ┬¦3 hold table ŌĆö that mechanism is deleted (see `docs/pipeline-architecture.md` ┬¦6).
2c. **sound-design** (Ōłź 2b ŌĆö depends on 2a + pedagogy, NOT on voice) ŌĆö writes `lesson-data/<id>/audio-cues.json`: bed mood, intro sting, outro resolve, and the SFXŌåÆbeat map. SEMANTIC, no frames. `toneSafe: true` for pinyin/tone lessons ŌåÆ flat pad bed + no melodic motif under narration. Follow `lesson-sound-design`. Music+SFX is a SECOND audio track that *consumes* the timeline; it does NOT drive cue length and is NOT frozen at 3a (see `docs/pipeline-architecture.md`).
3. **voice + ASR** Ōłź **primitive-gap-scan ŌåÆ primitive-builder** Ōłź **sound-asset gap-scan/factory** ŌĆö voice subagent runs `npm run lesson:voice`, verifies output, edits one cue's text + re-rolls if ASR matchScore is bad. Once accepted, the audio is **frozen** ŌĆö no further re-generation. Primitive lane reads visual-design + kids-eye + visual-discipline and names every visual demand the lesson makes. **Default answer is *compose existing primitives*.** A new primitive ships only when the gap is named explicitly. Builder produces test stills at real render size for the single hardest frame AND the worst-case multiplicity. The primitive node verifies its own test stills before the composer node consumes them. Sound-asset lane checks the curated library (`public/audio/_beds|_sfx|_stings/`) covers every key in `audio-cues.json` ŌĆö **default = REUSE**; a new bed/sting/SFX ships only when named, sourced author-time from CC0 freesound + Pixabay (per-asset `.license.txt`), and verified. Frozen before Wave 4.
3.5. **cue-timeline reconcile** (workflow node, mechanical) ŌĆö read `visualMotionSeconds` per cue from visual-design.md and narrationFrames per cue from `src/lessons/generated/<X>Timing.ts`. Compute `cueFrames = max(narrationFrames, motionFrames) + tailFrames` (tail Ōēż 9 frames / 0.3s). Chain cues end-to-end. Embed the reconciled cue list directly into `src/lessons/<X>LessonTimeline.ts` as the exported `<X>Cues` array. No `PADDED_CUE_DURATIONS_FRAMES`. Audio, visuals, and captions all read THIS timeline.
4. **composer** Ōłź **sketch-layer** ŌĆö both consume the **reconciled** cue boundaries. ZERO frame literals; every frame derives from `cues[id].startFrame + offset`. Composer never re-pads. If motion overruns its cue, cut non-load-bearing flourishes FIRST; then compress motion uniformly; never extend the cue. Composer reruns the pedagogy ┬¦1 audit against the rendered still before declaring done. Composer also wires the music: `<LessonBgmLayer>` in the Complete wrapper (MECHANICAL ŌĆö bed envelope reads cue narration windows + total) and `<LessonSfxLayer>` with SFX fired at the composer's OWN motion frames from `audio-cues.json` (bed = mechanical; SFX frames = composer-owned, derived from `cueFrames + layout.ts` offsets, never literals).
5. **render** ŌĆö `npm run lesson:render --skip-voice` (voice frozen from Wave 3). A post-render `ffmpeg loudnorm` pass normalizes the master to -16 LUFS / -1 dBTP (deterministic 2nd pass; `--skip-loudnorm` to skip during fast iterations).
6. **verification** ŌĆö reviews the rendered MP4 + contact sheet against pedagogy.md discoveries. The contact sheet is now action-aware (5 samples per cue: start / narr-mid / narr-end / hold-mid / cue-end). Stagnation is visible at a glance. Sound checks: melody NOT identifiable under narration; 3-point duck (intro duck / mid-gap rise / outro resolve); no SFX louder than narration. (The ŌłÆ16 LUFS / TP measurement is the LUFS gate in `lesson:check --measured`.)

Every subagent prompt must state input, output artifact, owned paths, and the lesson-agnostic rule.

Primitives live in `remotion-svg-primitives/src/shape-primitives/` (prop-driven, reusable). The composer consumes existing primitives and generated timing ŌĆö it does not create one-off SVG art inside lesson scenes.

Every lesson opens with a short text intro that announces the topic (lesson title + section + KP teaser). This is not optional and not a per-lesson design decision. Layout is a small composition of primitives, not a fixed template. If the normalized intro primitives don't yet exist (or the existing ones don't fit the new subject), the Wave 3 gap-scan node creates them ŌĆö designing the intro layout is the gap-scan's job, never the composer's. Once shipped they are reused by every subsequent lesson; only the per-lesson copy changes.

## Observability (per-node process logs)

The loop runs unattended and at scale, so every node must be inspectable after the fact. Three tiers, cheapest first:

1. **Structured return** (efficient digest) ŌĆö every node returns `{ status, outputArtifact, summary, issues, pipelineFindings }`; the workflow aggregates all returns into one final object: the whole run at a glance.
2. **Per-node log file** (`lesson-data/<id>/_logs/<wave>.md`, addressable by wave) ŌĆö each node writes it as its final action: INPUTS READ, OUTPUTS WRITTEN, COMMANDS RUN (+ exit status + key stdout/stderr, errors verbatim), KEY DECISIONS, ISSUES, PIPELINE FINDINGS.
3. **Raw transcript** (ground truth) ŌĆö the `Workflow` runtime persists each node's full `agent-<id>.jsonl` under the run's transcript dir: every tool call and output, for exact reproduction.

Read tier 1 first; drop to tier 2 for a wave that looks wrong; tier 3 only to reproduce a failure exactly. The union of all nodes' `pipelineFindings` is the workflow-improvement backlog.

## Discipline (every workflow node)

- **CUE IS THE UNIT OF COORDINATION.** Each cue has ONE timeline window shared by audio, visuals, and captions. The window is set by Wave 3.5 reconcile: `cueFrames = max(narrationFrames, visualMotionFrames) + tail`. If audio is on one timeline and visuals on another, that's a bug ŌĆö see `docs/pipeline-architecture.md` ┬¦1. NEVER re-introduce `PADDED_CUE_DURATIONS_FRAMES` or any timeline-padding table that the composer applies independently of audio.

- **AUDIO IS FROZEN AFTER WAVE 3a.** Once a cue's voice is accepted, the WAV is canonical. No re-generation for visual-fit reasons. If a cue's motion overruns, cut motion (Wave 4 trims flourishes first, then compresses); never re-record. This governs the NARRATION track only. Music + SFX are a SECOND track added at Wave 4 that *consumes* the reconciled timeline and changes no cue length ŌĆö adding it does not violate this freeze, and it is never an input to Wave 3.5 reconcile.

- **MEASURE, DON'T ASSUME.** Any code path that depends on voice output (cue lengths, motion budgets, sketch frame anchors) waits for actual voice + ASR. Estimates from audio-captions are hints for narration length only, not contract. Composer never runs before Wave 3 produces the real timing module AND Wave 3.5 produces the reconciled cues.

- **VOICE OUTPUT IS VERIFIED, NOT TRUSTED.** Wave 3a is a real subagent ŌĆö it owns running `npm run lesson:voice` AND verifying the result. Inside the subagent: read `gemini-voice.json` transcriptText vs script, walk per-cue matchScore + asrText + duration in the generated timing module, fix any timeline JSON it finds wrong (re-roll voice, adjust cue boundaries from ASR evidence, edit `script-cues.json` for ASR-friendliness and re-run). Hand back the verified timing module + a per-cue audit note (part of the node's `_logs/voice.md`). The run does not advance to reconcile/composer until that audit passes.

- **UPSTREAM FLAGS ARE NOT ADVISORY.** When audio-captions flags an ASR risk with a proposed fix, apply it or record an explicit decision to ignore with reason. Silent passes are forbidden.

- **BBOX MANIFEST IS A FAST PRE-FILTER; `--measured` IS GROUND TRUTH.** Every load-bearing element (those listed in visual-design ┬¦3 Visual Contract) registers in `src/lessons/<camelLessonId>/manifest.ts` with a `bboxAt(frame)` derived from the same constants in `src/lessons/<camelLessonId>/layout.ts` the scene uses, AND carries `{...measureProps("<sameId>")}` on its outermost `<g>` in the scene (the scene root calls `useMeasureHook()` once). The linear manifest uses LINEAR progress, so it UNDER-estimates any element whose easing overshoots (bouncy/spring) and samples only ~11 keyframes ŌĆö overshoot and between-keyframe overlaps are invisible to it. Before claiming Wave 4 done, the composer runs `npm run lesson:check -- --config lesson-data/<id>/pipeline.json --measured`: the fast linear pass still writes `summary.collisionCount`; the opt-in measured pass renders motion-peak frames, reads each element's TRUE `getBBox()`, and AUGMENTS `bbox-manifest.json` with a `measured` block (`summary.measuredCollisionCount`, `summary.gatesFailed`) plus the LUFS / caption-redundancy / contrast / legibility / motion gates. Review both counts + `contact-sheet.png` + any `primitive-checks/*.png`; the verification node re-checks them. Any non-zero collision (linear OR measured) and any failed gate requires a fix or an explicit written justification in the same response. The check is advisory, not blocking ŌĆö but a silent skip is forbidden, and a gate that cannot run must print `SKIP: <reason>`. Full spec: `docs/proposals/machine-gated-verification.md`.

- **PRIMITIVE QUALITY IS OWNED BY WAVE 3, NOT WAVE 4.** The composer self-grade covers pedagogy (does the frame teach what it should?) and composition correctness (layout, timing, motion). Primitive aesthetic quality ŌĆö does this thing look like what it claims to be, at real render size, in the hardest multiplicity? ŌĆö is the primitive-builder's responsibility, verified by the orchestrator at the Wave 3 ŌåÆ Wave 4 boundary against the test stills. A bad-looking primitive is a Wave 3 bug; kick back, don't patch in the scene.

- **ZERO FRAME LITERALS in scene code.** Every frame number in `src/lessons/<X>LessonScene.tsx` derives from `cues[id].startFrame + relativeOffset` or `cues[id].endFrame - relativeOffset`. Absolute master-timeline literals are a coordinate-space bug (same class as mixing client-rect pixels with viewBox coords).

- **ZERO RAW MOTION LITERALS in scene code.** Every easing curve and spring config in `src/lessons/*.tsx` reaches for a named export ŌĆö `EASE.*`, `SPRING.*` from `src/motion-primitives/curves.ts`. `<PopIn motion=...>` for entrance physics, `<TeacherMark boil=...>` for opt-in hand-drawn liveliness. Full reach guide and anti-patterns in `.agents/CAPABILITIES.md`.

## 3D effects ŌĆö decorative only

The lesson repo consumes `@studio/three-effects` (the kit at `~/Desktop/shared-3d/`) for 3D moments. Strict rules:

- **3D is decorative.** Topic-intro cards (`src/lessons/transitions/TopicIntroCard.tsx`) and section-handoff transitions (`SectionHandoff.tsx`). Nothing else.
- **Teaching primitives stay SVG.** No 3D-textured countables, no glass dots, no iridescent sticks. The teaching object the child is reasoning about is always SVG. 3D frames or punctuates a moment; it never *is* the teaching primitive.
- **All theme tokens come from `src/theme.ts`** via `kidsPreset` (`src/three-effects/preset-kids.ts`). Never inline hex in 3D code.
- **Frame discipline extends to 3D.** The same `cues[id].startFrame + offset` rule applies ŌĆö pass `entranceFrame={cues.foo.startFrame + 6}` into a 3D component, never an absolute literal.

Kids preset is the canonical kids 3D theme. The vlog repo (render-lab) used to have a kids preset; that one was deleted. If anyone reaches for kids in the vlog repo, they are wrong ŌĆö point them here.

## Skills

Use the matching skill at each step:

- `complete-video-pipeline` ŌĆö orchestrator overview, when starting any lesson
- `lesson-pedagogy` ŌĆö Wave 0 pedagogy gate: decide what the child discovers at each beat before any visual choice; loaded read-only by every downstream wave
- `lesson-storyboard` ŌĆö cue IDs, narration beats, required visuals
- `kids-eye` ŌĆö viewer-first discipline: measurement block + zones declaration + finger-cover test + identity-preserved transformation (required by visual-touching subagents)
- `visual-discipline` ŌĆö Visual Contract before choreography, one-metaphor rule, semantic groups, render-and-self-critique
- `early-childhood-visual-taste` ŌĆö palette, typography, motion vocabulary for early-math lessons
- `lesson-audio-captions` ŌĆö teacher script, captions, cue boundary intent (NOT absolute frames)
- `lesson-sound-design` ŌĆö Wave 2c: author `audio-cues.json` (bed mood, SFXŌåÆbeat map, tone-safe guard); semantic, no frames
- `sketch-explainer-layer` ŌĆö cue-relative teacher marks (no master-timeline absolutes)
- `remotion-lesson-composer` ŌĆö compose the lesson scene from approved artifacts; zero frame literals; emits `layout.ts` + `manifest.ts` (contract: `src/lessons/manifestTypes.ts`); runs `npm run lesson:check` before render-and-critique
- `lesson-verification` ŌĆö final check on rendered output; primary surface is `<lesson-id>-contact.png` (auto-built at end of `lesson:render`), cross-checked against `bbox-manifest.json` + `primitive-checks/*.png` and the MP4
- `lesson-debugger` ŌĆö invoked only after render, when the user reports feedback
- `three-effects-composition` ŌĆö kit recipe (Scene + Stage + Camera + Material + Geometry); use when adding a 3D moment to a lesson
- `three-effects-registry` ŌĆö kit growth ŌĆö only relevant when adding a new effect/material/camera to the kit itself

## Feedback & Debugging Loop

After render, when the user reviews the MP4 and reports an issue, invoke the `lesson-debugger` skill for triage. Do not load it during waves 1ŌĆō6.

Structural changes ŌĆö new skill, new wave, reordered waves, or changed subagent contracts ŌĆö require user approval before edits. Spec edits inside existing skills do not.

## Research ŌåÆ implementation logging

When you implement anything based on a `research/*.md` brief, append a `## Progress` section to that exact research file in the same change. Each entry records:

- Date (absolute, `YYYY-MM-DD`).
- Which research bullet(s) it addressed ŌĆö quote the line or reference the section heading.
- Where it landed ŌĆö `file:line`, capability id (`CAPABILITIES.md#<id>`), or skill name.
- Decisions to NOT bridge a gap, with one-line reason.

Why: the research brief is the audit trail for what we promised ourselves to act on. Without the progress log, future-us can't tell which findings were absorbed vs forgotten, and the same research gets re-run.

How to apply: any time a research-driven change ships (capability, skill edit, primitive tweak), edit the originating research file in the same commit. If the change spans multiple research files, log into each.
