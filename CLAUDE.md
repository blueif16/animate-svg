# Project Rules

## Entry Point

To make a lesson video, write **one file**: `lesson-data/<lesson-id>/brief.md`. Then run the pipeline. `brief.md` is the only lesson-specific input ‚Äî everything else (wave order, kids-eye blocks, frame-literal rules, voice audit, reporting) lives in CLAUDE.md and the skills.

**Brief schema.** Seven sections, all unique info; trim or extend as needed but keep it minimal. **Note:** the `Length` field is a HINT for scope-setting, not a contract. The actual lesson length emerges from Wave 3.5's reconcile (`sum(max(narration, motion) + tail)`). Brief targets that don't survive contact with real narration are accepted as drift ‚Äî see `docs/pipeline-architecture.md`.

```markdown
# <lesson-id>

**Audience.** <age, language>
**Length.** <seconds band ‚Äî HINT for scope, not a target the composer pads to>
**Builds on.** <prior lesson ids | none>
**Style.** <style-id | default>

## Knowledge point
<one paragraph: what the child leaves knowing>

## The one beat
<the central demonstration this video exists for ‚Äî concrete, visual>

## Out of scope
<what NOT to teach; boundary with adjacent KPs>

## Continuity
<reuse what from prior lessons; identity-invariant promises>

## Narration notes
<known TTS risks, precedent fixes ‚Äî optional>
```

**Naming rule.** Lesson IDs are kebab-case with a curriculum-position prefix so they never collide: `kp<n>-<semantic>` (e.g. `kp2-counting-by-tens`, `kp3-tens-and-ones-place`). The prefix flows everywhere ‚Äî folder, MP4, voice `.wav`, generated timing module, and React composition name (`CompleteKp2CountingByTensLesson`).

**Folder rule.** Inputs and generated outputs are split into two top-level locations, keyed by lesson id.

- **Inputs ‚Äî `lesson-data/<lesson-id>/`** (authored, hand-edited, version-controlled): `brief.md`, `pedagogy.md`, `storyboard.md`, `visual-design.md`, `audio-captions.md`, `sketch-overlay.md`, `primitive-gap-scan.md`, `verification.md`, `script-cues.json`, and the scaffolded `pipeline.json`. Nothing the pipeline writes lands here.
- **Outputs ‚Äî `out/<lesson-id>/`** (generated, derivable, can be `.gitignore`d): `<lesson-id>.mp4`, `<lesson-id>-contact.png` + `<lesson-id>-contact.json`, `gemini-voice.json`, `asr-alignment.json`, `bbox-manifest.json`, `primitive-checks/`, `verification-frames/`. `outDir` is derived mechanically from `lessonId` ‚Äî pipeline.json does not configure it per lesson.
- **Two technical exceptions**, dictated by Remotion/TS imports:
  - Voice `.wav` stays at `public/audio/<lesson-id>-voice.wav` ‚Äî Remotion serves static media from `public/`.
  - Generated audio modules stay at `src/lessons/generated/<camelLessonId>Clips.ts` (v4: the per-cue audio truth the reconcile chains) and `<camelLessonId>Timing.ts` (ASR ‚Äî QA only). Per-cue voice clips stay at `public/audio/<lesson-id>/clips/`. (Pre-v4 `<camelLessonId>Silences.ts` is legacy ‚Äî only fen-yu-he still imports it.) These are TypeScript modules / static media imported by scene code; the Remotion bundler resolves them via TS paths and moving them under `out/` breaks the build.

**Commands.**

```bash
cd remotion-svg-primitives && npm run lesson:scaffold -- --id <lesson-id>   # creates pipeline.json
cd remotion-svg-primitives && npm run lesson:render   -- --config lesson-data/<lesson-id>/pipeline.json
```

`pipeline.json` is mechanical (paths + voice config) and is generated by scaffold ‚Äî do not edit by hand. Reusable scripts, primitives, and components stay lesson-agnostic ‚Äî never hardcode lesson topics, copy, timings, or paths into them.

Use `-- --skip-voice` only when intentionally reusing an existing aligned voice file. (The render script auto-skips voice if the generated timing module already exists for the lesson.)

ASR alignment is an owned production artifact, not a pass/fail gate. Inspect the detected transcript, token events, timestamps, and tokenizer settings, then correct cue timing from evidence.

## Subagent Workflow (wave order)

**Authority.** `.claude/workflows/lesson-build.js` is the executable single source of truth for the wave order (and the discipline laws below) ‚Äî pi extracts the production prompts from it and cannot read this file. This section is the ambient Claude-Code summary; on any disagreement the workflow wins, so improve the chain by editing the workflow first.

The entire loop ‚Äî every wave below, **including pedagogy (Wave 0) and reconcile (Wave 3.5)** ‚Äî is owned by **one self-contained workflow** (the `Workflow` tool), parameterized by the lesson knowledge and runnable in parallel across many lessons. The orchestrator's only job is to spawn that workflow with the knowledge and observe; it implements nothing and owns no wave directly. Each wave is a clean-room subagent given concrete inputs. Invoking a single wave's subagent by hand is a **debug-only exception** ‚Äî taken at the *start* of a run while improving the system, never the default. See `docs/pipeline-architecture.md` for the timeline-coordination architecture (cue-as-unit, frozen audio, Wave 3.5 reconcile).

0. **pedagogy** (first workflow node) ‚Äî write `lesson-data/<id>/pedagogy.md` answering for each cue: what does the child discover here that they didn't know walking in? Follow `lesson-pedagogy` skill. No downstream wave advances without this file.
1. **storyboard** alone ‚Äî reads `pedagogy.md` + `.agents/TEACHING-ACTIONS.md`; tags each cue with its **teaching action(s)** (the teaching verb, decided before any layout), then cue IDs, narration beats, and required visuals (read off each move's `requires`). NO duration estimates.
2a. **visual-design** alone (SERIAL ‚Äî must finish before 2b) ‚Äî Visual Contract with zones, identity-invariant, **per-cue `visualMotionSeconds`** (the motion budget). Intent only, no absolute frames. Every row references a pedagogy discovery.
2b. **audio/captions** (depends on 2a) ‚Äî writes narration TO FIT each cue's visualMotionSeconds at the calibrated voice rate (~0.30s/Chinese-char for Aoede). NO ¬ß3 hold table ‚Äî that mechanism is deleted (see `docs/pipeline-architecture.md` ¬ß6).
2c. **sound-design** (‚à• 2b ‚Äî depends on 2a + pedagogy, NOT on voice) ‚Äî writes `lesson-data/<id>/audio-cues.json`: bed mood, intro sting, outro resolve, and the SFX‚Üíbeat map. SEMANTIC, no frames. `toneSafe: true` for pinyin/tone lessons ‚Üí flat pad bed + no melodic motif under narration. Follow `lesson-sound-design`. Music+SFX is a SECOND audio track that *consumes* the timeline; it does NOT drive cue length and is NOT frozen at 3a (see `docs/pipeline-architecture.md`).
3. **voice + ASR** ‚à• **primitive-gap-scan ‚Üí primitive-builder** ‚à• **sound-asset gap-scan/factory** ‚Äî voice subagent runs `npm run lesson:voice`, verifies output, edits one cue's text + re-rolls if ASR matchScore is bad. Once accepted, the audio is **frozen** ‚Äî no further re-generation. Primitive lane is **teaching-driven**: it reads the storyboard's teaching actions + `.agents/TEACHING-ACTIONS.md`, maps each move to its `requires` capability, and diffs against the catalog (not an already-drawn layout) to name every demand the lesson makes ‚Äî also reading visual-design + kids-eye + visual-discipline. **Default answer is *compose existing primitives*.** A new primitive ships only when the gap is named explicitly. Builder produces test stills at real render size for the single hardest frame AND the worst-case multiplicity. The primitive node verifies its own test stills before the composer node consumes them. Sound-asset lane checks the curated library (`public/audio/_beds|_sfx|_stings/`) covers every key in `audio-cues.json` ‚Äî **default = REUSE**; a new bed/sting/SFX ships only when named, sourced author-time from CC0 freesound + Pixabay (per-asset `.license.txt`), and verified. Frozen before Wave 4.
3.5. **cue-timeline reconcile** (workflow node, mechanical) ‚Äî read `visualMotionSeconds` per cue from visual-design.md and the per-cue clip module `src/lessons/generated/<X>Clips.ts` (exact narrationFrames + typed gap). Compute `cueFrames = max(narrationFrames + gapFrames, motionFrames) + tailFrames` (tail ‚â§ 9 frames / 0.3s) via the kit `reconcileClipTimeline`. Chain cues end-to-end. Embed the reconciled `<X>Cues` AND `<X>VoiceClips` arrays into `src/lessons/<X>LessonTimeline.ts`. No `PADDED_CUE_DURATIONS_FRAMES`, no ASR-boundary correction, no Silences import. Audio (per-cue clips), visuals, and captions all read THIS timeline.
4. **composer** ‚à• **sketch-layer** ‚Äî both consume the **reconciled** cue boundaries. ZERO frame literals; every frame derives from `cues[id].startFrame + offset`. Composer never re-pads. If motion overruns its cue, cut non-load-bearing flourishes FIRST; then compress motion uniformly; never extend the cue. Composer reruns the pedagogy ¬ß1 audit against the rendered still before declaring done. Composer also wires the music: `<LessonBgmLayer>` in the Complete wrapper (MECHANICAL ‚Äî bed envelope reads cue narration windows + total) and `<LessonSfxLayer>` with SFX fired at the composer's OWN motion frames from `audio-cues.json` (bed = mechanical; SFX frames = composer-owned, derived from `cueFrames + layout.ts` offsets, never literals).
5. **render** ‚Äî `npm run lesson:render --skip-voice` (voice frozen from Wave 3). A post-render `ffmpeg loudnorm` pass normalizes the master to -16 LUFS / -1 dBTP (deterministic 2nd pass; `--skip-loudnorm` to skip during fast iterations).
6. **verification** ‚Äî reviews the rendered MP4 + contact sheet against pedagogy.md discoveries. The contact sheet is now action-aware (5 samples per cue: start / narr-mid / narr-end / hold-mid / cue-end). Stagnation is visible at a glance. Sound checks: melody NOT identifiable under narration; 3-point duck (intro duck / mid-gap rise / outro resolve); no SFX louder than narration. (The ‚àí16 LUFS / TP measurement is the LUFS gate in `lesson:check --measured`.)

Every subagent prompt must state input, output artifact, owned paths, and the lesson-agnostic rule.

Primitives live in `remotion-svg-primitives/src/shape-primitives/` (prop-driven, reusable). The composer consumes existing primitives and generated timing ‚Äî it does not create one-off SVG art inside lesson scenes.

Every lesson opens with a short text intro that announces the topic (lesson title + section + KP teaser). This is not optional and not a per-lesson design decision. Layout is a small composition of primitives, not a fixed template. If the normalized intro primitives don't yet exist (or the existing ones don't fit the new subject), the Wave 3 gap-scan node creates them ‚Äî designing the intro layout is the gap-scan's job, never the composer's. Once shipped they are reused by every subsequent lesson; only the per-lesson copy changes.

## Observability (per-node process logs)

The loop runs unattended and at scale, so every node must be inspectable after the fact. Three tiers, cheapest first:

1. **Structured return** (efficient digest) ‚Äî every node returns `{ status, outputArtifact, summary, issues, pipelineFindings }`; the workflow aggregates all returns into one final object: the whole run at a glance.
2. **Per-node log file** (`lesson-data/<id>/_logs/<wave>.md`, addressable by wave) ‚Äî each node writes it as its final action: INPUTS READ, OUTPUTS WRITTEN, COMMANDS RUN (+ exit status + key stdout/stderr, errors verbatim), KEY DECISIONS, ISSUES, PIPELINE FINDINGS.
3. **Raw transcript** (ground truth) ‚Äî the `Workflow` runtime persists each node's full `agent-<id>.jsonl` under the run's transcript dir: every tool call and output, for exact reproduction.

Read tier 1 first; drop to tier 2 for a wave that looks wrong; tier 3 only to reproduce a failure exactly. The union of all nodes' `pipelineFindings` is the workflow-improvement backlog.

## Discipline (every workflow node)

**Authority.** These laws are mirrored here for ambient Claude-Code sessions; the executable source of truth is the `discipline()` preamble in `.claude/workflows/lesson-build.js` (inlined there because pi agents extract from it and never read this file). On any disagreement the workflow wins ‚Äî edit it first.

- **CUE IS THE UNIT OF COORDINATION ‚Äî TRUE FOR AUDIO TOO (v4 cue-anchored audio).** Each cue has ONE timeline window shared by audio, visuals, and captions, set by Wave 3.5 reconcile: `cueFrames = max(narrationFrames + gapFrames, visualMotionFrames) + tail`. Each cue owns its OWN measured voice clip, mounted in its own `<Sequence from={cue.startFrame}>` ‚Äî there is NO single continuous WAV, so a clip can NEVER cross a cue boundary and the old continuous-WAV drift (audio playing ahead of the picture) is structurally impossible. NEVER re-introduce `PADDED_CUE_DURATIONS_FRAMES`, a single continuous voice `<Sequence>`, or `detect-silences`-based boundary snapping ‚Äî see `docs/pipeline-architecture.md` ¬ß1 + v4 changelog.

- **AUDIO IS FROZEN AFTER WAVE 3a.** Once accepted, the per-cue voice CLIPS are canonical. No re-generation for visual-fit reasons. If a cue's motion overruns, cut motion (Wave 4 trims flourishes first, then compresses); never re-record. Each clip is TRIMMED of TTS padding at generation; its exact length IS the cue's narrationFrames. An intra-cue pause is a typed `gap` or sub-beats, NEVER an in-text ellipsis (`I'm‚Ä¶‚Ä¶ Sam` drones ‚Äî the audio gate flags it). This governs the NARRATION track only. Music + SFX are a SECOND track added at Wave 4 that *consumes* the reconciled timeline and changes no cue length ‚Äî adding it does not violate this freeze, and it is never an input to Wave 3.5 reconcile.

- **MEASURE, DON'T ASSUME.** Any code path that depends on voice output (cue lengths, motion budgets, sketch frame anchors) waits for actual voice + ASR. Estimates from audio-captions are hints for narration length only, not contract. Composer never runs before Wave 3 produces the real timing module AND Wave 3.5 produces the reconciled cues.

- **VOICE OUTPUT IS VERIFIED, NOT TRUSTED.** Wave 3a is a real subagent ‚Äî it owns running `npm run lesson:voice` AND verifying the result. Inside the subagent: read `gemini-voice.json` transcriptText vs script, walk per-cue matchScore + asrText + duration in the generated timing module, fix any timeline JSON it finds wrong (re-roll voice, adjust cue boundaries from ASR evidence, edit `script-cues.json` for ASR-friendliness and re-run). Hand back the verified timing module + a per-cue audit note (part of the node's `_logs/voice.md`). The run does not advance to reconcile/composer until that audit passes.

- **UPSTREAM FLAGS ARE NOT ADVISORY.** When audio-captions flags an ASR risk with a proposed fix, apply it or record an explicit decision to ignore with reason. Silent passes are forbidden.

- **MEASURED RENDER IS THE BBOX GROUND TRUTH; THE MANIFEST IS METADATA-ONLY.** Every load-bearing element (those listed in visual-design ¬ß3 Visual Contract) is declared in `src/lessons/<camelLessonId>/manifest.ts` as a bare `{ id, zone }` ‚Äî NO geometry, NO `bboxAt` (contract: `src/lessons/manifestTypes.ts`) ‚Äî AND carries `{...measureProps("<sameId>")}` on its outermost `<g>` in the scene (the scene root calls `useMeasureHook()` once). There is ONE source of geometry truth: `layout.ts` feeds the scene, and every box is read back off the render via `getBBox()` ‚Äî never hand-ported into a manifest. Before claiming Wave 4 done, the composer runs `npm run lesson:check -- --config lesson-data/<id>/pipeline.json` (the measured pass is THE bbox check and ALWAYS runs ‚Äî there is no fast-linear pre-filter and no opt-in: the legacy `--measured` flag is an accepted no-op). It bundles the scene, SSR-renders peak-aware sample frames (entrance-overshoot offsets layered on each cue's mid/late keyframes, so bouncy/spring overshoot is caught, not missed by linear sampling), reads each element's TRUE `getBBox()`, and AUGMENTS `bbox-manifest.json` with a `measured` block (`summary.measuredCollisionCount`, `summary.captionIntrusionCount`, `summary.gatesFailed`). Its gates split by CONFIDENCE and the set GROWS over time ‚Äî read `summary.gatesFailed` for what blocks today, never assume a fixed list: **structural, false-positive-free gates FAIL the run (exit 1)** ‚Äî the declared‚ü∫measured id bijection, and loudness on the delivered master MP4 ‚Äî so a composer/verifier self-check can never launder a failing lesson as green; **geometric / eye-checkable findings are advisory WARN** ‚Äî measured overlaps and caption-band intrusion have false-positive sources (opacity-blind crossfades, a label legitimately near a suppressed ribbon) and the human is the eye, so they surface loudly but do not by themselves exit 1. Review `summary.*` + `contact-sheet.png` + any `primitive-checks/*.png`; the verification node re-checks them. Any failed gate, and any measured collision or caption intrusion, requires a fix or an explicit written justification in the same response quoting the failing row (id + frame + value). A silent skip is forbidden ‚Äî a gate that cannot run must print `SKIP: <reason>` (a SKIP never fails the run); a tool crash exits 0 (a crash is not a gate failure). Full spec: `docs/proposals/machine-gated-verification.md`.

- **BOUNDING BOX = TRUE FOOTPRINT; DECORATION NEVER INSIDE A MEASURED GROUP.** A collision check is only as honest as each element's box, so three hygiene laws keep `getBBox` equal to the element's true footprint. (1) **A `measureProps(id)` `<g>` wraps ONLY the load-bearing teaching mark.** Decoration ‚Äî pulse/attention rings, glow halos, sparkles, eye-drawers, `<Breathe>`/shimmer wrappers used purely for liveliness ‚Äî lives OUTSIDE every measured group, as its OWN `zone:"decoration"` element (or untagged). Nesting decoration inside a load-bearing `<g>` inflates its box so it no longer matches the thing it claims to bound ‚Üí phantom or missed collisions. (Real case: `aggregator-prompt` measured 516√ó400 because a pulse ring sat inside the `labels` group; its honest label-only footprint was 960√ó60, and a phantom `labels:objects` collision with the dots followed ‚Äî the dots never touched anything.) (2) **Decoration is subordinate and must never sit on or cross a teaching mark.** The `decoration` zone is gate-exempt ONLY so a real background never false-flags ‚Äî it is NOT a license to park a ring on the dots. Decoration that overlaps teaching content is a defect: cut it (slim to the bone), do not exempt it. (3) **Scene measure-id ‚â° manifest element id (bijection).** Every `measureProps("x")` has exactly one `manifest.elements[].id === "x"` and vice-versa. A group tagged with an id the manifest never declares falls through to `decoration` (silently exempt); a manifest id never tagged in the scene is never measured. Both VOID detection. (Real case: the scene tagged one `"recap"` while the manifest declared `recap-beat-1-5/2-4/3-3` ‚Üí join failure, recap collision detection void.) **REVIEW SURFACE ‚Äî every still shown to a human carries its boxes.** Run `npm run lesson:bbox -- --config lesson-data/<id>/pipeline.json [--frames a,b]`: it draws each registered element's MEASURED `getBBox()` box (solid, colored by zone) with an `id ¬∑ zone ¬∑ W√óH` label, into `out/<id>/bbox-overlay/`. The manifest holds no geometry, so there is no declared box to diverge from ‚Äî the measured box IS the single truth: an inflated box (e.g. a decoration ring caught inside a load-bearing group) is the defect you SEE, and a box drawn RED marks a measured id the manifest never declared ‚Äî a bijection break the measured pass fails the run on. NEVER review a bare screenshot.

- **PRIMITIVE QUALITY IS OWNED BY WAVE 3, NOT WAVE 4.** The composer self-grade covers pedagogy (does the frame teach what it should?) and composition correctness (layout, timing, motion). Primitive aesthetic quality ‚Äî does this thing look like what it claims to be, at real render size, in the hardest multiplicity? ‚Äî is the primitive-builder's responsibility, verified by the orchestrator at the Wave 3 ‚Üí Wave 4 boundary against the test stills. A bad-looking primitive is a Wave 3 bug; kick back, don't patch in the scene.

- **ZERO FRAME LITERALS in scene code.** Every frame number in `src/lessons/<X>LessonScene.tsx` derives from `cues[id].startFrame + relativeOffset` or `cues[id].endFrame - relativeOffset`. Absolute master-timeline literals are a coordinate-space bug (same class as mixing client-rect pixels with viewBox coords).

- **SPOKEN ENUMERATION BINDS TO TOKEN ONSETS, never a step constant.** Any animation that marks a spoken count/sequence (count 1‚ÜíN, per-number point/circle, per-word pulse, read-along step) binds each step to that token's MEASURED ASR onset ‚Äî `cue.tokenOnsets` (cue-local, index-aligned to `targetTokens`) via the kit `stepFramesFromOnsets`/`tokenOnsetFrame` helper and the primitive's onset prop (`<OrderedRowSpotlight stepFrames=‚Ä¶>`). A fixed cadence (`SWEEP_STEP_FRAMES`, an even grid from the cue head) is the SAME class as a frame literal ‚Äî forbidden for spoken-synced stepping (the inter-word interval is voice-dependent and already measured). Constants stay fine for NON-spoken motion (pops, settles, breathe). Onsets unavailable ‚Üí fall back to the constant AND emit a `pipelineFinding`.

- **ZERO RAW MOTION LITERALS in scene code.** Every easing curve and spring config in `src/lessons/*.tsx` reaches for a named export ‚Äî `EASE.*`, `SPRING.*` from `src/motion-primitives/curves.ts`. `<PopIn motion=...>` for entrance physics, `<TeacherMark boil=...>` for opt-in hand-drawn liveliness. Full reach guide and anti-patterns in `.agents/CAPABILITIES.md`.

## Capabilities

The single registry of reusable craft tools the lesson pipeline ships with: `.agents/CAPABILITIES.md`. Read it before reaching for motion / sketch / primitive APIs in scene code. The same file documents the protocol for adding a new capability ‚Äî follow it whenever you ship a new reusable primitive, prop, or JSON config key.

Its **pedagogical twin** is `.agents/TEACHING-ACTIONS.md` ‚Äî the registry of teaching MOVES (what we teach *with*) and what each move *requires* (audio + visual/layout + component). Planning composes the lesson script from these moves *before* any layout; the Wave-3 gap-scan maps each move ‚Üí its required capability ‚Üí the catalog, so gaps are teaching-driven. Add a move via the protocol in that file.

## 3D effects ‚Äî decorative only

The lesson repo consumes `@studio/three-effects` (the kit at `~/Desktop/shared-3d/`) for 3D moments. Strict rules:

- **3D is decorative.** Topic-intro cards (`src/lessons/transitions/TopicIntroCard.tsx`) and section-handoff transitions (`SectionHandoff.tsx`). Nothing else.
- **Teaching primitives stay SVG.** No 3D-textured countables, no glass dots, no iridescent sticks. The teaching object the child is reasoning about is always SVG. 3D frames or punctuates a moment; it never *is* the teaching primitive.
- **All theme tokens come from `src/theme.ts`** via `kidsPreset` (`src/three-effects/preset-kids.ts`). Never inline hex in 3D code.
- **Frame discipline extends to 3D.** The same `cues[id].startFrame + offset` rule applies ‚Äî pass `entranceFrame={cues.foo.startFrame + 6}` into a 3D component, never an absolute literal.

Kids preset is the canonical kids 3D theme. The vlog repo (render-lab) used to have a kids preset; that one was deleted. If anyone reaches for kids in the vlog repo, they are wrong ‚Äî point them here.

## Generated decorative SVG assets

Fixed-form DECORATIVE objects (a roped bundle, a pointing hand, a mascot, a prop) are GENERATED, not hand-coded: describe ‚Üí Gemini image ‚Üí trace ‚Üí flat SVG. Pipeline is author-time, at `/Users/tk/Desktop/Omniscience/scripts/gen_asset_svg.py` (Gemini "Nano-Banana" via `google-genai` + `GOOGLE_API_KEY` ‚Üí flat-papercraft prompt ‚Üí potrace trace; the palette already = our `theme.ts`).

```bash
<omni>/agent/.venv/bin/python scripts/gen_asset_svg.py --name <kebab> --description "<object>" \
  --color-out remotion-svg-primitives/public/icons \
  --mono-out remotion-svg-primitives/public/icons/mono \
  --keep-png remotion-svg-primitives/out
cd remotion-svg-primitives && npm run icons:build   # regenerates src/shape-primitives/iconAssetData.ts
```

Render via `<IconAsset name="<kebab>" variant="color"|"mono" tint=... />`.

- **Visual-source order, ALWAYS:** reuse/compose existing primitives ‚Üí hand-code a PARAMETRIC primitive ‚Üí generate. (Procedural SVG-filter craft for representational objects is a low ceiling ‚Äî generation is the path.)
- **GENERATE only when ALL hold:** fixed form (no count/progress/state prop); decorative (NOT the teaching primitive the child reasons about ‚Äî same fence as 3D-decorative-only); recognizability is the bug at render size.
- **NEVER generate** countables, anything with a progress/count/state prop, identity-preserving morphs, or text/numerals ‚Äî those stay hand-coded SVG.
- Owned by Wave 3 primitive-builder (never the composer). Assets live at `public/icons/<name>.svg` (+ `mono/` + `.meta.json` provenance) and ARE committed (tiny + canonical).
- Recolor only via theme tokens ‚Äî SVG fills are `var(--icon-*, <hex>)` / mono `currentColor`, and `<IconAsset tint>` does this. NEVER hardcode hex that bypasses it.

## Skill-system stewardship

We are ALWAYS evolving our TRACKED skill systems at dev speed ‚Äî fix flaws the moment they're spotted (no timers, no batching). This is ambient: you do not need to be told each session. When you spot a flaw in a skill/markdown, a `pipelineFindings`-class issue recurs, **or the user gives feedback on an artifact**, run the **`hermes-skill-system`** skill to capture‚Üíroute‚Üíedit‚Üíverify‚Üícommit it as ONE atomic, revertible change. (Feedback on an MP4: triage symptom‚Üíwave with `lesson-debugger`, then when the flaw is systemic or recurring, encode the durable fix via `hermes-skill-system` ‚Äî don't just patch the one lesson.)

- **The registry is `.agents/tracked-systems.md`** ‚Äî the index OBSERVE reads. FOUR independent tracked systems, each with its own map + criteria + open-threads: **lesson-build** (the teaching flow; map `.agents/skill-system-map.md`) ¬∑ **capability-registry** (the drift-clean source of truth + item prose) ¬∑ **capability-gallery** (the browse/audit surface + `catalog-digest`) ¬∑ **capability-gap-filler** (the authoring factory). Read the registry first; open a system's map only when acting on it.
- **Scope ‚Äî what Hermes optimizes:** the TEXTUAL quality of skills / markdowns / registry-item prose. A fix may land in markdown OR a component code-snippet ‚Äî but the **orchestration/executor code is OUT** (`lesson-build.js`, `capability-gap-filler.js`, `pi-runner/` are wiring: tracked, not optimized here). Improve a wave by editing its SKILL or its declared hand-off contract; a flaw fixable ONLY in `.js`/`.mjs` is FLAGGED to the wiring owner, not edited under Hermes.
- **Routing:** a component-quality flaw surfaced in a lesson routes to a CAPABILITY system (registry/gallery/gap-filler), not lesson-build. A cross-cutting thread gets one owner + a `touches:` pointer, never double-logged.
- **Every edit must generalize across ALL future runs** ‚Äî never hard-code one case, never lean on reward-hackable tests; verification is an intent the next session/human checks, and the human is the eye for visuals.
- **`skillsys(<id>)` is STRICT ‚Äî it IS the Hermes self-iteration log, keep it PURE.** Use it ONLY for a genuine quality-iteration of a tracked system's own content (a SKILL rule, a registry item's prose, a component's code when the component itself is the flaw); `<id>` = the registry id, with why/lesson/verify in the body, one lesson per commit; `review-edits.sh` reads this log to judge whether edits generalize. **Systematic / infrastructure changes ‚Äî registry/map/criteria scaffolding, tooling, DEFINE setup ‚Äî do NOT carry `skillsys`**; they use ordinary `chore`/`docs`.
- **Two HITL checkpoints (per the Hermes OPERATE loop):** the human approves the concrete diff before a structural edit is committed, and the human decides whether to rerun the workflow to re-validate after an edit.

## Research ‚Üí implementation logging

When you implement anything based on a `research/*.md` brief, append a `## Progress` section to that exact research file in the same change. Each entry records:

- Date (absolute, `YYYY-MM-DD`).
- Which research bullet(s) it addressed ‚Äî quote the line or reference the section heading.
- Where it landed ‚Äî `file:line`, capability id (`CAPABILITIES.md#<id>`), or skill name.
- Decisions to NOT bridge a gap, with one-line reason.

Why: the research brief is the audit trail for what we promised ourselves to act on. Without the progress log, future-us can't tell which findings were absorbed vs forgotten, and the same research gets re-run.

How to apply: any time a research-driven change ships (capability, skill edit, primitive tweak), edit the originating research file in the same commit. If the change spans multiple research files, log into each.

## Production executor ‚Äî pi agents (`pi-runner/`)

The lesson-build workflow has TWO executors. **Dev:** Claude Code's Workflow tool (`.claude/workflows/lesson-build.js`). **Production:** a fleet of **pi agents** (pi.dev / `earendil-works/pi`) running cheap non-Claude coding-plan models, driven by `pi-runner/`. The workflow is deterministic, so the split is fixed: a plain-code **driver owns the graph** (stage order, parallel lanes, status); **pi is the per-node executor** (one `pi -p` per wave). Nodes coordinate via the filesystem exactly as the waves do.

- **`lesson-build.js` is the SINGLE SOURCE OF TRUTH ‚Äî never hand-sync pi.** pi-runner does not re-define waves: `pi-runner/extract.mjs` *executes* `lesson-build.js` under recording stubs (the Workflow hooks) to derive the exact prompts + DAG (execute-and-record). So authoring/proving the Claude Code Workflow is the ONLY edit; pi runs identical prompts automatically ‚Äî no drift, no codegen, all 14 waves + skill refs included. Skill *content* edits already flow to both (nodes read `SKILL.md` by path).

- **ALWAYS the orchestrator runs everything; the user runs NOTHING.** Claude Code executes every `pi-runner` command itself and checks status itself ‚Äî never hand the user a command, never ask whether a run is still alive. You are expected to know the exact live status at all times.
- **ALWAYS run with `--debug`.** Debug mode prints a 4s heartbeat (elapsed ¬∑ events ¬∑ tool count ¬∑ last event ¬∑ Œî-since-last-event ¬∑ stall flag), refreshes `out/<id>/run-status.json` continuously, and hard-kills a node after `--node-timeout` (default 600s) so a hang is visible in seconds, never minutes. Production mode is lean (10s status refresh, no console heartbeat) for the unattended fleet. Status is VERIFIED ‚Äî a node is `ok` only if its declared artifacts exist on disk.
- **Model + credential live ONCE in pi's own global config** `~/.pi/agent/models.json` (providers `cp` ‚Äî DashScope-intl OpenAI-compatible, qwen3.7-max ‚Äî and `minimax` ‚Äî MiniMax-M3) ‚Äî pi resolves it natively for EVERY product, so no per-repo key, no `.env` credential, no provider extension. The pi DEFAULT for this repo is now `minimax` / `MiniMax-M3` (set via `pi-runner/.env` wiring: `PI_RUNNER_PROVIDER`/`PI_RUNNER_MODEL`); `cp` / qwen3.7-max remains available as a non-default (`pi --provider cp`). `pi-runner/.env` is wiring-only. Swap the default = edit that one file; verify `pi --list-models <provider>`.
- **Headless pi invariants** (a silent ~10-min startup hang taught these): close stdin, pass `--offline` and `--no-extensions` (the explicit `-e` provider extension still loads). Full spec: `pi-runner/README.md`.
- Run: `node pi-runner/run.mjs --lesson <id> --until <wave> --debug`. Fleet: one driver per lesson in the background; poll each `run-status.json` for only the important info.
