# Project Rules

## Entry Point

To make a lesson video, write **one file**: `lesson-data/<lesson-id>/brief.md`. Then run the pipeline. `brief.md` is the only lesson-specific input — everything else (wave order, kids-eye blocks, frame-literal rules, voice audit, reporting) lives in CLAUDE.md and the skills.

**Brief schema.** Seven sections, all unique info; trim or extend as needed but keep it minimal. **Note:** the `Length` field is a HINT for scope-setting, not a contract. The actual lesson length emerges from Wave 3.5's reconcile (`sum(max(narration, motion) + tail)`). Brief targets that don't survive contact with real narration are accepted as drift — see `docs/pipeline-architecture.md`.

```markdown
# <lesson-id>

**Audience.** <age, language>
**Length.** <seconds band — HINT for scope, not a target the composer pads to>
**Builds on.** <prior lesson ids | none>
**Style.** <style-id | default>

## Knowledge point
<one paragraph: what the child leaves knowing>

## The one beat
<the central demonstration this video exists for — concrete, visual>

## Out of scope
<what NOT to teach; boundary with adjacent KPs>

## Continuity
<reuse what from prior lessons; identity-invariant promises>

## Narration notes
<known TTS risks, precedent fixes — optional>
```

**Naming rule.** Lesson IDs are kebab-case with a curriculum-position prefix so they never collide: `kp<n>-<semantic>` (e.g. `kp2-counting-by-tens`, `kp3-tens-and-ones-place`). The prefix flows everywhere — folder, MP4, voice `.wav`, generated timing module, and React composition name (`CompleteKp2CountingByTensLesson`).

**Folder rule.** Inputs and generated outputs are split into two top-level locations, keyed by lesson id.

- **Inputs — `lesson-data/<lesson-id>/`** (authored, hand-edited, version-controlled): `brief.md`, `pedagogy.md`, `storyboard.md`, `visual-design.md`, `audio-captions.md`, `sketch-overlay.md`, `primitive-gap-scan.md`, `verification.md`, `script-cues.json`, and the scaffolded `pipeline.json`. Nothing the pipeline writes lands here.
- **Outputs — `out/<lesson-id>/`** (generated, derivable, can be `.gitignore`d): `<lesson-id>.mp4`, `<lesson-id>-contact.png` + `<lesson-id>-contact.json`, `gemini-voice.json`, `asr-alignment.json`, `bbox-manifest.json`, `primitive-checks/`, `verification-frames/`. `outDir` is derived mechanically from `lessonId` — pipeline.json does not configure it per lesson.
- **Two technical exceptions**, dictated by Remotion/TS imports:
  - Voice `.wav` stays at `public/audio/<lesson-id>-voice.wav` — Remotion serves static media from `public/`.
  - Generated audio modules stay at `src/lessons/generated/<camelLessonId>Clips.ts` (v4: the per-cue audio truth the reconcile chains) and `<camelLessonId>Timing.ts` (ASR — QA only). Per-cue voice clips stay at `public/audio/<lesson-id>/clips/`. (Pre-v4 `<camelLessonId>Silences.ts` is legacy — only fen-yu-he still imports it.) These are TypeScript modules / static media imported by scene code; the Remotion bundler resolves them via TS paths and moving them under `out/` breaks the build.

**Commands.**

```bash
cd remotion-svg-primitives && npm run lesson:scaffold -- --id <lesson-id>   # creates pipeline.json
cd remotion-svg-primitives && npm run lesson:render   -- --config lesson-data/<lesson-id>/pipeline.json
```

`pipeline.json` is mechanical (paths + voice config) and is generated by scaffold — do not edit by hand. Reusable scripts, primitives, and components stay lesson-agnostic — never hardcode lesson topics, copy, timings, or paths into them.

Use `-- --skip-voice` only when intentionally reusing an existing aligned voice file. (The render script auto-skips voice if the generated timing module already exists for the lesson.)

ASR alignment is an owned production artifact, not a pass/fail gate. Inspect the detected transcript, token events, timestamps, and tokenizer settings, then correct cue timing from evidence.

## Subagent Workflow (wave order)

**Authority.** `.claude/workflows/lesson-build.js` is the executable single source of truth for the wave order (and the discipline laws below) — pi extracts the production prompts from it and cannot read this file. This section is the ambient Claude-Code summary; on any disagreement the workflow wins, so improve the chain by editing the workflow first.

The entire loop — every wave below, **including pedagogy (Wave 0) and reconcile (Wave 3.5)** — is owned by **one self-contained workflow** (the `Workflow` tool), parameterized by the lesson knowledge and runnable in parallel across many lessons. The orchestrator's only job is to spawn that workflow with the knowledge and observe; it implements nothing and owns no wave directly. Each wave is a clean-room subagent given concrete inputs. Invoking a single wave's subagent by hand is a **debug-only exception** — taken at the *start* of a run while improving the system, never the default. See `docs/pipeline-architecture.md` for the timeline-coordination architecture (cue-as-unit, frozen audio, Wave 3.5 reconcile).

0. **pedagogy** (first workflow node) — write `lesson-data/<id>/pedagogy.md` answering for each cue: what does the child discover here that they didn't know walking in? Follow `lesson-pedagogy` skill. No downstream wave advances without this file.
1. **storyboard** alone — reads `pedagogy.md` + `.agents/TEACHING-ACTIONS.md`; tags each cue with its **teaching action(s)** (the teaching verb, decided before any layout), then cue IDs, narration beats, and required visuals (read off each move's `requires`). NO duration estimates.
2a. **visual-design** alone (SERIAL — must finish before 2b) — Visual Contract with zones, identity-invariant, **per-cue `visualMotionSeconds`** (the motion budget). Intent only, no absolute frames. Every row references a pedagogy discovery.
2b. **audio/captions** (depends on 2a) — writes narration TO FIT each cue's visualMotionSeconds at the calibrated voice rate (~0.30s/Chinese-char for Aoede). NO §3 hold table — that mechanism is deleted (see `docs/pipeline-architecture.md` §6).
2c. **sound-design** (∥ 2b — depends on 2a + pedagogy, NOT on voice) — writes `lesson-data/<id>/audio-cues.json`: bed mood, intro sting, outro resolve, and the SFX→beat map. SEMANTIC, no frames. `toneSafe: true` for pinyin/tone lessons → flat pad bed + no melodic motif under narration. Follow `lesson-sound-design`. Music+SFX is a SECOND audio track that *consumes* the timeline; it does NOT drive cue length and is NOT frozen at 3a (see `docs/pipeline-architecture.md`).
3. **voice + ASR** ∥ **primitive-gap-scan → primitive-builder** ∥ **sound-asset gap-scan/factory** — voice subagent runs `npm run lesson:voice`, verifies output, edits one cue's text + re-rolls if ASR matchScore is bad. Once accepted, the audio is **frozen** — no further re-generation. Primitive lane is **teaching-driven**: it reads the storyboard's teaching actions + `.agents/TEACHING-ACTIONS.md`, maps each move to its `requires` capability, and diffs against the catalog (not an already-drawn layout) to name every demand the lesson makes — also reading visual-design + kids-eye + visual-discipline. **Default answer is *compose existing primitives*.** A new primitive ships only when the gap is named explicitly. Builder produces test stills at real render size for the single hardest frame AND the worst-case multiplicity. The primitive node verifies its own test stills before the composer node consumes them. Sound-asset lane checks the curated library (`public/audio/_beds|_sfx|_stings/`) covers every key in `audio-cues.json` — **default = REUSE**; a new bed/sting/SFX ships only when named, sourced author-time from CC0 freesound + Pixabay (per-asset `.license.txt`), and verified. Frozen before Wave 4.
3.5. **cue-timeline reconcile** (workflow node, mechanical) — read `visualMotionSeconds` per cue from visual-design.md and the per-cue clip module `src/lessons/generated/<X>Clips.ts` (exact narrationFrames + typed gap). Compute `cueFrames = max(narrationFrames + gapFrames, motionFrames) + tailFrames` (tail ≤ 9 frames / 0.3s) via the kit `reconcileClipTimeline`. Chain cues end-to-end. Embed the reconciled `<X>Cues` AND `<X>VoiceClips` arrays into `src/lessons/<X>LessonTimeline.ts`. No `PADDED_CUE_DURATIONS_FRAMES`, no ASR-boundary correction, no Silences import. Audio (per-cue clips), visuals, and captions all read THIS timeline.
4. **composer** ∥ **sketch-layer** — both consume the **reconciled** cue boundaries. ZERO frame literals; every frame derives from `cues[id].startFrame + offset`. Composer never re-pads. If motion overruns its cue, cut non-load-bearing flourishes FIRST; then compress motion uniformly; never extend the cue. Composer reruns the pedagogy §1 audit against the rendered still before declaring done. Composer also wires the music: `<LessonBgmLayer>` in the Complete wrapper (MECHANICAL — bed envelope reads cue narration windows + total) and `<LessonSfxLayer>` with SFX fired at the composer's OWN motion frames from `audio-cues.json` (bed = mechanical; SFX frames = composer-owned, derived from `cueFrames + layout.ts` offsets, never literals).
5. **render** — `npm run lesson:render --skip-voice` (voice frozen from Wave 3). A post-render `ffmpeg loudnorm` pass normalizes the master to -16 LUFS / -1 dBTP (deterministic 2nd pass; `--skip-loudnorm` to skip during fast iterations).
6. **verification** — reviews the rendered MP4 + contact sheet against pedagogy.md discoveries. The contact sheet is now action-aware (5 samples per cue: start / narr-mid / narr-end / hold-mid / cue-end). Stagnation is visible at a glance. Sound checks: melody NOT identifiable under narration; 3-point duck (intro duck / mid-gap rise / outro resolve); no SFX louder than narration. (The −16 LUFS / TP measurement is the LUFS gate in `lesson:check --measured`.)

Every subagent prompt must state input, output artifact, owned paths, and the lesson-agnostic rule.

Primitives live in `remotion-svg-primitives/src/shape-primitives/` (prop-driven, reusable). The composer consumes existing primitives and generated timing — it does not create one-off SVG art inside lesson scenes.

Every lesson opens with a short text intro that announces the topic (lesson title + section + KP teaser). This is not optional and not a per-lesson design decision. Layout is a small composition of primitives, not a fixed template. If the normalized intro primitives don't yet exist (or the existing ones don't fit the new subject), the Wave 3 gap-scan node creates them — designing the intro layout is the gap-scan's job, never the composer's. Once shipped they are reused by every subsequent lesson; only the per-lesson copy changes.

## Observability (per-node process logs)

The loop runs unattended and at scale, so every node must be inspectable after the fact. Three tiers, cheapest first:

1. **Structured return** (efficient digest) — every node returns `{ status, outputArtifact, summary, issues, pipelineFindings }`; the workflow aggregates all returns into one final object: the whole run at a glance.
2. **Per-node log file** (`lesson-data/<id>/_logs/<wave>.md`, addressable by wave) — each node writes it as its final action: INPUTS READ, OUTPUTS WRITTEN, COMMANDS RUN (+ exit status + key stdout/stderr, errors verbatim), KEY DECISIONS, ISSUES, PIPELINE FINDINGS.
3. **Raw transcript** (ground truth) — the `Workflow` runtime persists each node's full `agent-<id>.jsonl` under the run's transcript dir: every tool call and output, for exact reproduction.

Read tier 1 first; drop to tier 2 for a wave that looks wrong; tier 3 only to reproduce a failure exactly. The union of all nodes' `pipelineFindings` is the workflow-improvement backlog.

## Discipline (every workflow node)

**Authority.** These laws are mirrored here for ambient Claude-Code sessions; the executable source of truth is the `discipline()` preamble in `.claude/workflows/lesson-build.js` (inlined there because pi agents extract from it and never read this file). On any disagreement the workflow wins — edit it first.

- **CUE IS THE UNIT OF COORDINATION — TRUE FOR AUDIO TOO (v4 cue-anchored audio).** Each cue has ONE timeline window shared by audio, visuals, and captions, set by Wave 3.5 reconcile: `cueFrames = max(narrationFrames + gapFrames, visualMotionFrames) + tail`. Each cue owns its OWN measured voice clip, mounted in its own `<Sequence from={cue.startFrame}>` — there is NO single continuous WAV, so a clip can NEVER cross a cue boundary and the old continuous-WAV drift (audio playing ahead of the picture) is structurally impossible. NEVER re-introduce `PADDED_CUE_DURATIONS_FRAMES`, a single continuous voice `<Sequence>`, or `detect-silences`-based boundary snapping — see `docs/pipeline-architecture.md` §1 + v4 changelog.

- **AUDIO IS FROZEN AFTER WAVE 3a.** Once accepted, the per-cue voice CLIPS are canonical. No re-generation for visual-fit reasons. If a cue's motion overruns, cut motion (Wave 4 trims flourishes first, then compresses); never re-record. Each clip is TRIMMED of TTS padding at generation; its exact length IS the cue's narrationFrames. An intra-cue pause is a typed `gap` or sub-beats, NEVER an in-text ellipsis (`I'm…… Sam` drones — the audio gate flags it). This governs the NARRATION track only. Music + SFX are a SECOND track added at Wave 4 that *consumes* the reconciled timeline and changes no cue length — adding it does not violate this freeze, and it is never an input to Wave 3.5 reconcile.

- **MEASURE, DON'T ASSUME.** Any code path that depends on voice output (cue lengths, motion budgets, sketch frame anchors) waits for actual voice + ASR. Estimates from audio-captions are hints for narration length only, not contract. Composer never runs before Wave 3 produces the real timing module AND Wave 3.5 produces the reconciled cues.

- **VOICE OUTPUT IS VERIFIED, NOT TRUSTED.** Wave 3a is a real subagent — it owns running `npm run lesson:voice` AND verifying the result. Inside the subagent: read `gemini-voice.json` transcriptText vs script, walk per-cue matchScore + asrText + duration in the generated timing module, fix any timeline JSON it finds wrong (re-roll voice, adjust cue boundaries from ASR evidence, edit `script-cues.json` for ASR-friendliness and re-run). Hand back the verified timing module + a per-cue audit note (part of the node's `_logs/voice.md`). The run does not advance to reconcile/composer until that audit passes.

- **UPSTREAM FLAGS ARE NOT ADVISORY.** When audio-captions flags an ASR risk with a proposed fix, apply it or record an explicit decision to ignore with reason. Silent passes are forbidden.

- **BBOX MANIFEST IS A FAST PRE-FILTER; `--measured` IS GROUND TRUTH.** Every load-bearing element (those listed in visual-design §3 Visual Contract) registers in `src/lessons/<camelLessonId>/manifest.ts` with a `bboxAt(frame)` derived from the same constants in `src/lessons/<camelLessonId>/layout.ts` the scene uses, AND carries `{...measureProps("<sameId>")}` on its outermost `<g>` in the scene (the scene root calls `useMeasureHook()` once). The linear manifest uses LINEAR progress, so it UNDER-estimates any element whose easing overshoots (bouncy/spring) and samples only ~11 keyframes — overshoot and between-keyframe overlaps are invisible to it. Before claiming Wave 4 done, the composer runs `npm run lesson:check -- --config lesson-data/<id>/pipeline.json --measured`: the fast linear pass still writes `summary.collisionCount`; the opt-in measured pass renders motion-peak frames, reads each element's TRUE `getBBox()`, and AUGMENTS `bbox-manifest.json` with a `measured` block (`summary.measuredCollisionCount`, `summary.gatesFailed`) plus the LUFS / caption-redundancy / contrast / legibility / motion gates. Review both counts + `contact-sheet.png` + any `primitive-checks/*.png`; the verification node re-checks them. Any non-zero collision (linear OR measured) and any failed gate requires a fix or an explicit written justification in the same response. The check is advisory, not blocking — but a silent skip is forbidden, and a gate that cannot run must print `SKIP: <reason>`. Full spec: `docs/proposals/machine-gated-verification.md`.

- **PRIMITIVE QUALITY IS OWNED BY WAVE 3, NOT WAVE 4.** The composer self-grade covers pedagogy (does the frame teach what it should?) and composition correctness (layout, timing, motion). Primitive aesthetic quality — does this thing look like what it claims to be, at real render size, in the hardest multiplicity? — is the primitive-builder's responsibility, verified by the orchestrator at the Wave 3 → Wave 4 boundary against the test stills. A bad-looking primitive is a Wave 3 bug; kick back, don't patch in the scene.

- **ZERO FRAME LITERALS in scene code.** Every frame number in `src/lessons/<X>LessonScene.tsx` derives from `cues[id].startFrame + relativeOffset` or `cues[id].endFrame - relativeOffset`. Absolute master-timeline literals are a coordinate-space bug (same class as mixing client-rect pixels with viewBox coords).

- **SPOKEN ENUMERATION BINDS TO TOKEN ONSETS, never a step constant.** Any animation that marks a spoken count/sequence (count 1→N, per-number point/circle, per-word pulse, read-along step) binds each step to that token's MEASURED ASR onset — `cue.tokenOnsets` (cue-local, index-aligned to `targetTokens`) via the kit `stepFramesFromOnsets`/`tokenOnsetFrame` helper and the primitive's onset prop (`<OrderedRowSpotlight stepFrames=…>`). A fixed cadence (`SWEEP_STEP_FRAMES`, an even grid from the cue head) is the SAME class as a frame literal — forbidden for spoken-synced stepping (the inter-word interval is voice-dependent and already measured). Constants stay fine for NON-spoken motion (pops, settles, breathe). Onsets unavailable → fall back to the constant AND emit a `pipelineFinding`.

- **ZERO RAW MOTION LITERALS in scene code.** Every easing curve and spring config in `src/lessons/*.tsx` reaches for a named export — `EASE.*`, `SPRING.*` from `src/motion-primitives/curves.ts`. `<PopIn motion=...>` for entrance physics, `<TeacherMark boil=...>` for opt-in hand-drawn liveliness. Full reach guide and anti-patterns in `.agents/CAPABILITIES.md`.

## Capabilities

## 3D effects — decorative only

The lesson repo consumes `@studio/three-effects` (the kit at `~/Desktop/shared-3d/`) for 3D moments. Strict rules:

- **3D is decorative.** Topic-intro cards (`src/lessons/transitions/TopicIntroCard.tsx`) and section-handoff transitions (`SectionHandoff.tsx`). Nothing else.
- **Teaching primitives stay SVG.** No 3D-textured countables, no glass dots, no iridescent sticks. The teaching object the child is reasoning about is always SVG. 3D frames or punctuates a moment; it never *is* the teaching primitive.
- **All theme tokens come from `src/theme.ts`** via `kidsPreset` (`src/three-effects/preset-kids.ts`). Never inline hex in 3D code.
- **Frame discipline extends to 3D.** The same `cues[id].startFrame + offset` rule applies — pass `entranceFrame={cues.foo.startFrame + 6}` into a 3D component, never an absolute literal.

Kids preset is the canonical kids 3D theme. The vlog repo (render-lab) used to have a kids preset; that one was deleted. If anyone reaches for kids in the vlog repo, they are wrong — point them here.

## Generated decorative SVG assets

Fixed-form DECORATIVE objects (a roped bundle, a pointing hand, a mascot, a prop) are GENERATED, not hand-coded: describe → Gemini image → trace → flat SVG. Pipeline is author-time, at `/Users/tk/Desktop/Omniscience/scripts/gen_asset_svg.py` (Gemini "Nano-Banana" via `google-genai` + `GOOGLE_API_KEY` → flat-papercraft prompt → potrace trace; the palette already = our `theme.ts`).

```bash
<omni>/agent/.venv/bin/python scripts/gen_asset_svg.py --name <kebab> --description "<object>" \
  --color-out remotion-svg-primitives/public/icons \
  --mono-out remotion-svg-primitives/public/icons/mono \
  --keep-png remotion-svg-primitives/out
cd remotion-svg-primitives && npm run icons:build   # regenerates src/shape-primitives/iconAssetData.ts
```

Render via `<IconAsset name="<kebab>" variant="color"|"mono" tint=... />`.

- **Visual-source order, ALWAYS:** reuse/compose existing primitives → hand-code a PARAMETRIC primitive → generate. (Procedural SVG-filter craft for representational objects is a low ceiling — generation is the path.)
- **GENERATE only when ALL hold:** fixed form (no count/progress/state prop); decorative (NOT the teaching primitive the child reasons about — same fence as 3D-decorative-only); recognizability is the bug at render size.
- **NEVER generate** countables, anything with a progress/count/state prop, identity-preserving morphs, or text/numerals — those stay hand-coded SVG.
- Owned by Wave 3 primitive-builder (never the composer). Assets live at `public/icons/<name>.svg` (+ `mono/` + `.meta.json` provenance) and ARE committed (tiny + canonical).
- Recolor only via theme tokens — SVG fills are `var(--icon-*, <hex>)` / mono `currentColor`, and `<IconAsset tint>` does this. NEVER hardcode hex that bypasses it.

## Skill-system stewardship

We are ALWAYS evolving this skill system at dev speed — fix flaws the moment they're spotted (no timers, no batching). This is ambient: you do not need to be told each session. When you spot a flaw in a skill or in the workflow, a `pipelineFindings`-class issue recurs, **or the user gives feedback on a rendered artifact**, run the **`hermes-skill-system`** skill to capture→route→edit→verify→commit it as ONE atomic, revertible change. (Feedback on an MP4: triage symptom→wave with `lesson-debugger`, then when the flaw is systemic or recurring, encode the durable fix via `hermes-skill-system` — don't just patch the one lesson.)

- **The map is `.agents/skill-system-map.md`** — every skill + the `lesson-build.js` orchestrator that wires them + each owner's responsibility + where each run's logs/artifacts live + a diagnostics log. Read it (with the run's real evidence) to find where a flaw belongs; refresh it via the skill's INIT when the system's shape changes. It gets more certain with every run.
- **Prefer fixing the workflow/orchestrator over a single skill** when the flaw is coordination/hand-off (`lesson-build.js`: "improve a wave by editing its SKILL; improve the chain by editing this file").
- **Every edit must generalize across ALL future runs** — never hard-code one case, never lean on reward-hackable tests; verification is an intent the next session/human checks, and the human is the eye for visuals.
- **One commit per lesson:** `skillsys(<owner>): <rule>` with why/lesson/verify in the body. Review a span with the skill's `review-edits.sh`. (A new global skill itself is a structural change — but the per-lesson `skillsys` edits it produces are normal spec edits, not structural changes.)
- **Two HITL checkpoints (per the Hermes OPERATE loop):** the human approves the concrete diff before a structural edit is committed, and the human decides whether to rerun the workflow to re-validate after an edit.

## Feedback & Debugging Loop

After render, when the user reviews the MP4 and reports an issue, invoke the `lesson-debugger` skill for triage. Do not load it during waves 1–6.

Structural changes — new skill, new wave, reordered waves, or changed subagent contracts — require user approval before edits. Spec edits inside existing skills do not.

## Research → implementation logging

When you implement anything based on a `research/*.md` brief, append a `## Progress` section to that exact research file in the same change. Each entry records:

- Date (absolute, `YYYY-MM-DD`).
- Which research bullet(s) it addressed — quote the line or reference the section heading.
- Where it landed — `file:line`, capability id (`CAPABILITIES.md#<id>`), or skill name.
- Decisions to NOT bridge a gap, with one-line reason.

Why: the research brief is the audit trail for what we promised ourselves to act on. Without the progress log, future-us can't tell which findings were absorbed vs forgotten, and the same research gets re-run.

How to apply: any time a research-driven change ships (capability, skill edit, primitive tweak), edit the originating research file in the same commit. If the change spans multiple research files, log into each.

## Production executor — pi agents (`pi-runner/`)

The lesson-build workflow has TWO executors. **Dev:** Claude Code's Workflow tool (`.claude/workflows/lesson-build.js`). **Production:** a fleet of **pi agents** (pi.dev / `earendil-works/pi`) running cheap non-Claude coding-plan models, driven by `pi-runner/`. The workflow is deterministic, so the split is fixed: a plain-code **driver owns the graph** (stage order, parallel lanes, status); **pi is the per-node executor** (one `pi -p` per wave). Nodes coordinate via the filesystem exactly as the waves do.

- **`lesson-build.js` is the SINGLE SOURCE OF TRUTH — never hand-sync pi.** pi-runner does not re-define waves: `pi-runner/extract.mjs` *executes* `lesson-build.js` under recording stubs (the Workflow hooks) to derive the exact prompts + DAG (execute-and-record). So authoring/proving the Claude Code Workflow is the ONLY edit; pi runs identical prompts automatically — no drift, no codegen, all 14 waves + skill refs included. Skill *content* edits already flow to both (nodes read `SKILL.md` by path).

- **ALWAYS the orchestrator runs everything; the user runs NOTHING.** Claude Code executes every `pi-runner` command itself and checks status itself — never hand the user a command, never ask whether a run is still alive. You are expected to know the exact live status at all times.
- **ALWAYS run with `--debug`.** Debug mode prints a 4s heartbeat (elapsed · events · tool count · last event · Δ-since-last-event · stall flag), refreshes `out/<id>/run-status.json` continuously, and hard-kills a node after `--node-timeout` (default 600s) so a hang is visible in seconds, never minutes. Production mode is lean (10s status refresh, no console heartbeat) for the unattended fleet. Status is VERIFIED — a node is `ok` only if its declared artifacts exist on disk.
- **Model + credential live ONCE in pi's own global config** `~/.pi/agent/models.json` (providers `cp` — DashScope-intl OpenAI-compatible, qwen3.7-max — and `minimax` — MiniMax-M3) — pi resolves it natively for EVERY product, so no per-repo key, no `.env` credential, no provider extension. The pi DEFAULT for this repo is now `minimax` / `MiniMax-M3` (set via `pi-runner/.env` wiring: `PI_RUNNER_PROVIDER`/`PI_RUNNER_MODEL`); `cp` / qwen3.7-max remains available as a non-default (`pi --provider cp`). `pi-runner/.env` is wiring-only. Swap the default = edit that one file; verify `pi --list-models <provider>`.
- **Headless pi invariants** (a silent ~10-min startup hang taught these): close stdin, pass `--offline` and `--no-extensions` (the explicit `-e` provider extension still loads). Full spec: `pi-runner/README.md`.
- Run: `node pi-runner/run.mjs --lesson <id> --until <wave> --debug`. Fleet: one driver per lesson in the background; poll each `run-status.json` for only the important info.

This is the canonical surface for **browsing and auditing the component library** — use it before building a lesson (what exists to REUSE) and when auditing for decorative-vs-load-bearing components.
